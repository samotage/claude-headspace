# Channel Trust Models

## Overview

Claude Headspace enforces two distinct trust models for channel communication, architecturally separated by how agent identity is verified. This document describes each model and the security boundaries between them.

## Internal Agents (Infrastructure Identity)

Internal agents run in tmux sessions managed by Claude Headspace. Their identity is verified through infrastructure — the tmux pane ID bound to an Agent record in the database.

### How identity works

1. Agent is created by the system and assigned a `tmux_pane_id`
2. The `caller_identity.resolve_caller()` function detects the current tmux pane via `tmux display-message`
3. The pane ID is looked up against active Agent records
4. If a match is found, the agent's identity is confirmed

### What internal agents CAN do

- Produce work output that is relayed to channels via system-mediated routing
- Read channel history via read-only CLI commands (`flask channel list`, `flask channel show`, `flask channel members`, `flask msg history`)
- Read persona information via `flask persona list`, `flask persona handoffs`

### What internal agents CANNOT do

- Post messages directly to channels via HTTP endpoints (the `X-Headspace-Agent-ID` header path has been removed)
- Post messages via CLI (`flask msg send` is restricted to operators)
- Create, modify, or manage channels via CLI (all mutating `flask channel` commands are restricted to operators)
- Register personas via CLI (`flask persona register` is restricted to operators)
- Set the `HEADSPACE_AGENT_ID` env var to impersonate another agent (the env var override has been removed from `resolve_caller()`)

### System-mediated routing

Internal agent output reaches channels only through `ChannelDeliveryService.relay_agent_response()`, which:

1. Is called by `HookReceiver.process_stop()` when a Turn is classified
2. Only relays turns with `COMPLETION` or `END_OF_COMMAND` intents
3. Posts the message via `ChannelService.send_message()` with full provenance (source_turn_id, source_command_id)
4. Strips the `COMMAND COMPLETE` footer before relay

This ensures the system controls what agent output enters channels, preventing agents from injecting arbitrary messages.

## Remote Agents (Validated Token Identity)

Remote agents are created via the Remote Agents API and authenticated using session tokens.

### How identity works

1. Remote agent is created via `POST /api/remote_agents/create`
2. A session token is generated and returned to the caller
3. All subsequent API calls include `Authorization: Bearer <token>`
4. The `SessionTokenService` validates the token and resolves the agent

### What remote agents CAN do

- Post messages to channels via `POST /api/channels/<slug>/messages` with a valid Bearer token
- Perform all channel operations available through the REST API (create, join, leave, etc.)
- Their identity is cryptographically bound to their session token

### Security properties

- Tokens are opaque, generated by `SessionTokenService`
- Token validation is per-agent scoped
- Invalid or expired tokens result in `401 invalid_session_token`

## Operator (Session Cookie Identity)

Dashboard operators authenticate via Flask session cookies.

### How identity works

1. Operator accesses the dashboard via browser
2. Flask session cookie is set
3. `Persona.get_operator()` resolves the operator persona from the session

### What operators CAN do

- All channel operations via REST API and CLI
- All mutating CLI commands (create, add, leave, complete, transfer-chair, mute, unmute, send)
- Persona registration via `flask persona register`

## Security Boundaries Summary

| Capability | Internal Agent | Remote Agent | Operator |
|---|---|---|---|
| Direct HTTP channel posting | No | Yes (Bearer token) | Yes (session cookie) |
| CLI `flask msg send` | No (rejected) | N/A | Yes |
| CLI mutating channel commands | No (rejected) | N/A | Yes |
| CLI read-only commands | Yes | N/A | Yes |
| System-mediated relay | Yes (COMPLETION/END_OF_COMMAND only) | No | No |
| `flask persona register` | No (rejected) | N/A | Yes |

## Design Principles

1. **Security by capability removal.** No new tokens, secrets, or authentication mechanisms were added. The spoofable paths (HTTP header, env var) were removed entirely.

2. **Infrastructure-verified identity.** Internal agent identity comes from tmux pane binding, which cannot be spoofed by the agent itself.

3. **CLI restriction via caller-context detection.** Mutating CLI commands detect agent context via `reject_if_agent_context()` in `cli_utils.py`, which uses `resolve_caller()` — if resolution succeeds, the caller is an agent and the command is rejected.
