## Executive Summary

Channels exist as database tables (Sprint 3) but have no business logic or user-facing interface. This sprint creates the ChannelService â€” the single service class that all channel operations flow through â€” and the CLI layer that agents use to interact with channels from their terminal sessions.

The ChannelService is registered as `app.extensions["channel_service"]` and owns all channel operations: creation, membership management, message writes, lifecycle transitions, and capability checks. The CLI (`flask channel` + `flask msg`) is a thin Click wrapper that resolves the calling agent's identity, delegates to ChannelService methods, and formats output. The same service backs the API endpoints (S5), voice bridge (S8), and dashboard (S7) â€” those sprints add frontends, not logic.

Message sending is fire-and-forget: the service writes the Message record to the database and returns immediately. Fan-out delivery to other channel members is Sprint 6's concern. System messages (joins, leaves, state transitions) are generated by the service, not exposed via CLI or API.

All design decisions are resolved in the Inter-Agent Communication Workshop, Section 2 (Decisions 2.1â€“2.2). The data model from Section 1 (Decisions 1.1â€“1.5) provides the structural foundation. See `docs/workshop/interagent-communication/sections/section-2-channel-operations.md`.

## 1. Context & Purpose

### 1.1 Context

Sprint 3 created the Channel, ChannelMembership, and Message tables with their relationships, constraints, and enum types. The database can store channels, memberships, and messages â€” but nothing creates, queries, or validates them. There is no service layer to enforce lifecycle rules, no capability checks, no CLI for agents to use.

Agents interact with the system via bash tools â€” `flask` CLI commands are their primary interface. The operator uses the dashboard, voice bridge, and API. All of these need to call the same service layer. Without a centralised ChannelService, each frontend would implement its own validation, state transitions, and error handling â€” a recipe for inconsistency.

The existing codebase follows a clear pattern: services are classes registered in `app.extensions`, CLI commands are Click/AppGroup wrappers that call service methods, and routes are thin HTTP wrappers over the same services. This sprint follows that pattern exactly.

### 1.2 Target User

**Agents (CLI consumers):** Internal agent-type personas running in tmux sessions. They create channels, add members, send messages, and read history via `flask channel` and `flask msg` commands.

**The operator (indirect):** The ChannelService methods are also called by the API (S5), voice bridge (S8), and dashboard (S7). This sprint builds the service; those sprints build the frontends.

### 1.3 Success Moment

Robbo needs to discuss a data model decision with Con. From the terminal, Robbo runs `flask channel create "model-review" --type review --members con`. A channel is created, Robbo is chair, Con is added (if Con has no running agent, Headspace spins one up). Robbo sends a message: `flask msg send review-model-review-7 "The persona_id constraint needs to be nullable â€” here's why..."`. Con's agent receives the message via tmux (Sprint 6), reads context via `flask msg history review-model-review-7`, and responds. The entire interaction happens through CLI commands that agents can invoke as bash tools.

## 3. Success Criteria

### 3.1 Functional Success Criteria

1. `flask channel create "test-channel" --type workshop` creates a channel with slug `workshop-test-channel-{id}`, status `pending`, and the calling agent's persona as chair
2. `flask channel list` returns only channels where the calling persona is an active member
3. `flask channel list --all` returns all non-archived channels (operator-equivalent visibility)
4. `flask channel show <slug>` displays channel details including name, type, status, description, members, message count, and timestamps
5. `flask channel add <slug> --persona <persona-slug>` adds a persona to the channel and posts a system message
6. If the added persona has no running agent, the system initiates async agent spin-up and returns immediately with confirmation
7. `flask channel leave <slug>` sets membership status to `left` and posts a system message
8. When the last active member leaves a channel, the channel auto-transitions to `complete`
9. `flask channel complete <slug>` transitions the channel to `complete` state (chair or operator only)
10. `flask channel transfer-chair <slug> --to <persona-slug>` transfers chair role (current chair only)
11. `flask channel mute <slug>` sets membership status to `muted`; `flask channel unmute <slug>` sets it back to `active`
12. `flask msg send <slug> <content>` writes a Message record and returns immediately
13. Sending the first non-system message to a `pending` channel transitions it to `active`
14. `flask msg history <slug>` displays messages in conversational envelope format
15. `flask msg history <slug> --limit 20 --since <ISO>` supports pagination
16. An agent attempting to join a second active channel receives an actionable error with the current channel name and a leave command
17. A persona without channel creation capability receives a clear error on `flask channel create`
18. Non-chair personas receive a clear error on `flask channel complete` and `flask channel transfer-chair`
19. Non-members receive a clear error on `flask msg send` and `flask msg history`
20. Caller identity resolution works via `HEADSPACE_AGENT_ID` env var override (takes precedence) with tmux pane detection as the primary fallback

### 3.2 Non-Functional Success Criteria

1. ChannelService is the single point of truth for all channel operations â€” no business logic in CLI commands or route handlers
2. All service methods are independently testable without CLI or HTTP context
3. CLI commands produce machine-parseable output where appropriate (`--format yaml` on history)
4. System messages are generated exclusively by the service layer, never exposed as CLI-callable message types
5. The service handles concurrent operations safely (advisory locks where needed, database constraints as backstop)

## 4. Functional Requirements (FRs)

### ChannelService Class

**FR1: Service registration**
The `ChannelService` class shall be instantiated in the app factory and registered as `app.extensions["channel_service"]`. It receives a reference to the Flask app for accessing other services and database sessions.

**FR2: Channel creation**
`create_channel(creator_persona, name, channel_type, description=None, intent_override=None, organisation_id=None, project_id=None, member_slugs=None)` shall:
- Validate the creator persona has channel creation capability
- Create the Channel record with status `pending`
- Create a ChannelMembership for the creator with `is_chair=True`
- If `member_slugs` provided, add each persona as a member (same logic as FR6)
- Post a system message for each member addition
- Return the created Channel

**FR3: Channel listing**
`list_channels(persona, status=None, channel_type=None, all_visible=False)` shall return channels filtered by the calling persona's active memberships. With `all_visible=True`, return all non-archived channels regardless of membership.

**FR4: Channel details**
`get_channel(slug)` shall return the Channel with its members and message count, or raise an error if not found.

**FR4a: Channel update**
`update_channel(slug, persona, description=None, intent_override=None)` shall validate the caller is the chair or operator, update the specified fields, and return the updated Channel. Only `description` and `intent_override` are mutable. Fail with `NotChairError` if the caller is neither the chair nor the operator.

**FR5: Channel completion**
`complete_channel(slug, persona)` shall validate the caller is the chair, transition status to `complete`, set `completed_at`, and post a system message. Fail with a clear error if the caller is not the chair.

### Membership Management

**FR5b: List members**
`list_members(slug)` shall return all ChannelMembership records for the channel, including persona name, agent_id, status (active/left/muted), `is_chair`, and `joined_at`/`left_at`. This provides a richer result than `get_channel().memberships` by including online/offline agent status.

**FR6: Add member**
`add_member(slug, persona_slug, caller_persona)` shall:
- Validate the caller is an active member of the channel
- Validate the target persona exists and is not already a member
- Validate the channel is not complete or archived
- Find or spin up an agent for the target persona (async spin-up if no running agent â€” same creation + readiness polling as remote agents)
- Enforce the one-agent-one-channel constraint with an actionable error
- Create the ChannelMembership record
- Post a system message ("{persona_name} joined the channel")
- Generate a context briefing (last 10 messages) for the new member

**FR7: Leave channel**
`leave_channel(slug, persona)` shall set membership status to `left`, set `left_at`, and post a system message. If the last active member leaves (muted members do not count as active), auto-transition the channel to `complete`.

**FR8: Chair transfer**
`transfer_chair(slug, target_persona_slug, caller_persona)` shall validate the caller is the current chair, validate the target is an active member, clear `is_chair` on the current chair, set `is_chair` on the target, and post a system message.

**FR9: Mute/unmute**
`mute_channel(slug, persona)` shall set membership status to `muted`. `unmute_channel(slug, persona)` shall set it back to `active`. Both post system messages.

### Message Operations

**FR10: Send message**
`send_message(slug, content, persona, agent=None, message_type='message', attachment_path=None, source_turn_id=None, source_command_id=None)` shall:
- Validate the sender is an active member of the channel
- Validate the channel is not complete or archived
- Validate `message_type` is `message`, `delegation`, or `escalation` (not `system` â€” service-generated only)
- Write the Message record
- If this is the first non-system message and channel status is `pending`, transition to `active`
- Return the created Message

**FR11: Message history**
`get_history(slug, persona, limit=50, since=None, before=None)` shall:
- Validate the caller is a member (active, left, or muted â€” left members can still read history)
- Return messages ordered by `sent_at` ascending, with cursor pagination via `since` and `before` ISO timestamps
- Include persona name, agent ID, content, message type, and sent_at for each message

### Capability Checks

**FR12: Channel creation capability**
`create_channel()` shall check `creator_persona.can_create_channel` (property from S2 FR7) before creating a channel. If `False`, raise `NoCreationCapabilityError`.

**FR13: System messages**
System messages shall be generated exclusively by the service for joins, leaves, state changes, and chair transfers. The `system` message type is not callable via CLI or API. System messages have `persona_id=NULL` (no sender persona).

**FR14: Agent-to-membership linking on agent registration**
When the session correlator links a new agent to a persona (via `_create_or_update_agent()`), the system shall check for any ChannelMembership where `persona_id = agent.persona_id AND agent_id IS NULL AND status = 'active'` and update `agent_id` to the new agent's ID. This ensures agents spun up via `add_member` are linked to their channel membership when they register.

**FR14a: Context briefing delivery on post-registration**
After FR14 links an agent to a ChannelMembership, the system shall generate and deliver the context briefing (last 10 messages) to the agent via tmux. This is the delivery trigger for agents spun up asynchronously by `add_member` â€” the briefing is generated and delivered at registration time (when the agent is ready), not at add time (when the agent may not exist). If the channel has no messages, no briefing is delivered.

**FR15: Archive channel**
`archive_channel(slug, persona)` shall validate the caller is the chair or operator, transition status to `archived`, set `archived_at`, and post a system message. Fail with a clear error if the caller is not the chair or operator, or if the channel is not in `complete` state.

### CLI Commands

**FR16: `flask channel` command group**
A Click AppGroup named `channel` registered in the app factory. All subcommands resolve caller identity before delegating to ChannelService.

**FR17: `flask msg` command group**
A Click AppGroup named `msg` registered in the app factory. Subcommands: `send` and `history`.

**FR18: Caller identity resolution**
All CLI commands that require caller identity shall use a two-strategy cascade:
1. Override: `HEADSPACE_AGENT_ID` env var â€” takes precedence when set
2. Primary: `tmux display-message -p '#{pane_id}'` -> look up Agent by `tmux_pane_id`
If neither resolves, fail with: `Error: Cannot identify calling agent. Are you running in a Headspace-managed session?`

## 5. Non-Functional Requirements (NFRs)

**NFR1: Service isolation**
ChannelService shall have no dependency on CLI or HTTP layers. It operates on model objects and returns model objects or raises typed exceptions. CLI and API wrap it with I/O formatting.

**NFR2: Error hierarchy**
The service shall define a `ChannelError` base exception with subclasses for specific failure modes: `ChannelNotFoundError`, `NotAMemberError`, `NotChairError`, `ChannelClosedError`, `AlreadyMemberError`, `NoCreationCapabilityError`, `AgentChannelConflictError`. Each carries a human-readable message suitable for direct display. Note: `CallerResolutionError` is defined separately in `services/caller_identity.py` as a standalone exception (not a `ChannelError` subclass) because caller identity resolution is shared infrastructure used by CLI commands, not channel-specific logic.

**NFR3: Database safety**
All state-modifying operations shall use explicit transactions. Advisory locks shall protect operations where concurrent access could violate business rules (e.g., chair transfer, last-member-leave auto-complete). Database constraints (unique indexes, foreign keys) serve as a backstop, not the primary enforcement mechanism.

**NFR4: Service pattern consistency**
ChannelService follows the same `__init__(self, app)` pattern as `HandoffExecutor`, `HeadspaceMonitor`, `PriorityScoringService`, and other services that need app context for accessing extensions and database sessions.

**NFR5: CLI output consistency**
CLI commands use `click.echo()` for output. Error messages go to stderr via `click.echo(..., err=True)`. Success messages are concise. Table formatting follows the pattern in `persona_cli.py`.

### 6.5 Caller Identity Resolution

Shared utility used by both CLI groups. Two-strategy cascade:

```python
import os
import subprocess

from ..database import db
from ..models.agent import Agent


class CallerResolutionError(Exception):
    """Cannot identify calling agent."""


def resolve_caller() -> Agent:
    """Resolve the calling agent via env var override or tmux pane detection.

    Strategy 1 (override): HEADSPACE_AGENT_ID env var â€” takes precedence when set.
    Strategy 2 (primary): tmux display-message pane detection.
    """
    # Strategy 1: env var override (takes precedence when set)
    agent_id_str = os.environ.get("HEADSPACE_AGENT_ID")
    if agent_id_str:
        try:
            agent_id = int(agent_id_str)
            agent = Agent.query.filter_by(id=agent_id, ended_at=None).first()
            if agent:
                return agent
        except ValueError:
            pass

    # Strategy 2: tmux pane detection
    try:
        result = subprocess.run(
            ["tmux", "display-message", "-p", "#{pane_id}"],
            capture_output=True, text=True, timeout=5
        )
        if result.returncode == 0:
            pane_id = result.stdout.strip()
            if pane_id:
                agent = Agent.query.filter_by(
                    tmux_pane_id=pane_id, ended_at=None
                ).first()
                if agent:
                    return agent
    except (subprocess.TimeoutExpired, FileNotFoundError):
        pass

    raise CallerResolutionError(
        "Error: Cannot identify calling agent. "
        "Are you running in a Headspace-managed session?"
    )
```

### 6.8 Conversational Envelope Format

Default output format for `flask msg history`. Each message rendered as:

```
[#channel-slug] PersonaName (agent:ID) -- DD Mon YYYY, HH:MM:
Message content here, potentially multiple lines.
Markdown preserved as-is.

[#channel-slug] PersonaName (agent:ID) -- DD Mon YYYY, HH:MM:
Next message content.
```

System messages use a different header format:

```
[#channel-slug] SYSTEM -- DD Mon YYYY, HH:MM:
Paula joined the channel.
```

For person-type personas (no agent ID):

```
[#channel-slug] Sam -- DD Mon YYYY, HH:MM:
Message from the operator.
```

Implementation: iterate over Message records, format the header line, print content, separate with blank line.

The `--format yaml` option outputs machine-consumable YAML:

```yaml
- id: 42
  channel_slug: workshop-model-review-7
  persona_slug: architect-robbo-3
  persona_name: Robbo
  agent_id: 1103
  content: "The persona_id constraint is resolved."
  message_type: message
  sent_at: "2026-03-03T10:23:45Z"
```

### 6.9 Capability Check â€” `can_create_channel`

### 6.10 One-Agent-One-Channel Enforcement

The partial unique index `uq_active_agent_one_channel` on `channel_memberships (agent_id) WHERE status = 'active' AND agent_id IS NOT NULL` (created in S3) is the database backstop. The service layer checks proactively to return an actionable error:

```python
def _check_agent_channel_conflict(self, agent: Agent) -> None:
    """Check if the agent is already in an active channel."""
    if agent is None:
        return
    existing = ChannelMembership.query.filter_by(
        agent_id=agent.id, status="active"
    ).first()
    if existing:
        channel = existing.channel
        raise AgentChannelConflictError(
            f"Error: Agent #{agent.id} ({agent.persona.name}) is already "
            f"an active member of #{channel.slug}.\n"
            f"Leave that channel first: flask channel leave {channel.slug}"
        )
```

This check runs during `add_member` before creating the membership record. The database constraint catches any race condition the service check misses.

### 6.11 Context Briefing on Member Add

When a persona is added to a channel that already has messages, the service generates a context briefing â€” the last 10 messages formatted as a single text block. This briefing is injected into the new agent's tmux session after persona skill injection.

```python
def _generate_context_briefing(self, channel: Channel, limit: int = 10) -> str:
    """Generate a context briefing from recent channel messages.

    Returns a formatted text block with the last N messages,
    suitable for injection into an agent's tmux session.
    """
    messages = (
        Message.query
        .filter_by(channel_id=channel.id)
        .order_by(Message.sent_at.desc())
        .limit(limit)
        .all()
    )
    if not messages:
        return ""

    messages.reverse()  # Chronological order

    lines = [f"=== Context briefing: #{channel.slug} (last {len(messages)} messages) ===\n"]
    for msg in messages:
        sender = msg.persona.name if msg.persona else "SYSTEM"
        timestamp = msg.sent_at.strftime("%d %b %Y, %H:%M")
        lines.append(f"[{sender}] ({timestamp}): {msg.content}\n")
    lines.append("=== End of context briefing ===")

    return "\n".join(lines)
```

Delivered as a single synthesised block, not replayed as individual messages (per Decision 2.1).

### 6.12 Channel Status Transitions

The service manages all status transitions. No direct status column writes from CLI or API.

| From | To | Trigger | Service method |
|------|----|---------|----------------|
| `pending` | `active` | First non-system message sent | `_transition_to_active()` called from `send_message()` |
| `active` | `complete` | Chair calls `complete_channel()` | `complete_channel()` |
| `active` | `complete` | Last active member leaves (muted members excluded) | `_auto_complete_if_empty()` called from `leave_channel()` |
| `complete` | `archived` | Chair or operator calls `archive_channel()` | `archive_channel()` |

No reactivation. `ChannelClosedError` raised on any operation against a `complete` or `archived` channel. Error: `Error: Channel #channel-slug is complete. Create a new channel to continue.`

### 6.13 Agent Spin-Up on Member Add

When `add_member` targets a persona with no running agent, the service uses the existing remote agent creation pattern (same `create_agent` function used by `RemoteAgentService` and `HandoffExecutor`):

```python
def _spin_up_agent_for_persona(self, persona: Persona) -> Agent | None:
    """Find or create an agent for a persona.

    Returns the active agent, or None if spin-up fails.
    If no active agent exists, creates one using the same
    pattern as remote agent and handoff successor creation.
    """
    # Check for existing active agent
    active_agent = Agent.query.filter_by(
        persona_id=persona.id, ended_at=None
    ).first()
    if active_agent:
        return active_agent

    # No active agent â€” spin up
    from .agent_lifecycle import create_agent
    result = create_agent(persona_slug=persona.slug)
    if not result.success:
        logger.warning(
            f"Agent spin-up failed for persona {persona.slug}: {result.message}"
        )
        return None
    # Agent creation is async â€” agent_id not immediately available
    # Return None and let the membership be created with agent_id=NULL
    # The agent will be linked when it registers via session-start hook
    return None
```

The CLI returns immediately: `Added {persona_name} to #{slug}. Agent spinning up...` The membership record is created with `agent_id=NULL`. When the agent starts and registers via the session-start hook, the session correlator links it to the persona and updates the ChannelMembership's `agent_id` (see FR14).

### 6.18 Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Caller identity resolution fails in non-tmux environments (testing, CI) | Medium | Low | `HEADSPACE_AGENT_ID` env var provides a reliable fallback. Tests set the env var directly. |
| Agent spin-up on member add creates orphaned agents if the channel is completed before the agent starts | Low | Low | The agent registers normally via session-start hook. Orphaned agents are cleaned by the existing agent reaper (5-min timeout). |
| Race condition on last-member-leave auto-complete when two members leave simultaneously | Very Low | Low | Database advisory lock on the channel ID during leave operations. The partial unique index prevents duplicate active memberships. |
| `can_create_channel` is too permissive for v1 (all internal personas can create) | Medium | Low | Intentional for v1. Tightening requires persona type awareness which S2 provides. Service method is the single point of change. |
