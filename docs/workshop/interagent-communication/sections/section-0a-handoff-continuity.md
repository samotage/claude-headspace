# Section 0A: Handoff Continuity & Synthetic Injection

**Status:** Fully resolved (7 decisions: 0A.1–0A.7)
**Workshop:** [Epic 9 — Inter-Agent Communication](../interagent-communication-workshop.md)
**Depends on:** None (independent of Sections 1–5)
**Canonical data model:** [`../../erds/headspace-org-erd-full.md`](../../erds/headspace-org-erd-full.md) — resolved outcomes are stored there, not in embedded ERDs below.

**Purpose:** Design the agent lifecycle continuity system — startup detection of prior handoffs, operator-gated rehydration, and on-demand handoff history. Introduces the "synthetic injection" delivery primitive: a software-generated precursor turn injected into the agent's conversation by the application, distinct from both hook responses and tmux send-keys.

**Added:** 2 March 2026, based on workshop session between operator (Sam) and architect (Robbo).

**Relationship to channels:** This section is independent of the channel model (Sections 1–4) and can be delivered as a standalone sprint. However, the synthetic injection mechanism introduced here may inform channel message delivery design. The channel sections should reference this primitive when considering how software-generated messages enter an agent's context.

---

### Design Context (from workshop session)

The operator identified three gaps in the current handoff system:

1. **No startup detection:** When a new agent starts, it has no awareness of prior handoffs for its persona. The current system pushes handoff context via tmux injection — the agent is told what to read. If the successor was created by the handoff executor, it gets the injection. But if an agent is started manually (operator fires up a new Robbo session), there's no way to know that Robbo's predecessor left work behind.

2. **No scannable handoff index:** The current filename format (`{YYYYMMDDTHHmmss}-{agent-8digit}.md`) is opaque. You can't tell what a handoff is about without opening the file. The directory listing is useless as an index.

3. **No operator gate:** The current system auto-injects handoff context into the successor's conversation. The operator has no say in whether the agent should continue from a handoff or start fresh. The injection consumes context window whether or not the work is still relevant.

**Operator's design direction:**
- Detection is **deterministic** — software scans the filesystem, not the agent
- Presentation is a **synthetic injection** — a software-generated precursor turn, not tmux
- Rehydration is **operator-gated** — the operator decides, then tells the agent to read a specific file
- Summary is **baked into the filename** — the departing agent writes the summary as part of the handoff instruction
- Scope is **by persona** — handoff directory lives under `data/personas/{slug}/handoffs/`

### 0A.1 Handoff Document Filename Format
- [x] **Decision: What filename format makes the handoff directory a scannable index?**

**Depends on:** None

**Context:** The current format is `{YYYYMMDDTHHmmss}-{agent-8digit}.md`. The operator's direction is to include a summary description in the filename so the directory listing itself becomes the handoff history.

**Resolution:**

#### Format: `{timestamp}_{summary-slug}_{agent-id:NNN}.md`

Three sections separated by underscores:

| Section | Format | Generated by |
|---|---|---|
| Timestamp | ISO 8601: `YYYY-MM-DDTHH:MM:SS` | Software (pre-filled) |
| Summary slug | Kebab-case description, max 60 chars | Departing agent |
| Agent ID | Literal `agent-id:` prefix + integer ID (no zero-padding) | Software (pre-filled) |

**Example:** `2026-03-01T10:23:32_advisory-locking-implementation-complete_agent-id:342.md`

**Current format (legacy):** `20260302T233900-00001137.md` — opaque, can't tell what the handoff is about without opening it.

#### Summary constraints

- Kebab-case (lowercase, hyphens between words)
- Max 60 characters
- No underscores (reserved as section separator)
- Generated by the departing agent as part of the handoff instruction

#### Legacy handoffs

Accept mixed formats. Old-format files still sort by timestamp (the prefix is still a date). No migration — the startup detection (0A.2) handles both formats by sorting directory contents by filename prefix.

#### No DB summary column

The filename IS the summary. The Handoff DB model already carries `file_path` — the summary is parseable from the filename. No new column needed.

### 0A.2 Startup Detection Mechanism
- [x] **Decision: How does the software detect available handoffs when an agent starts?**

**Depends on:** 0A.1 (filename format must be settled to parse summaries)

**Context:** When a new agent is created and associated with a persona, Headspace should scan the persona's handoff directory for existing handoff documents and surface the most recent ones.

**Resolution:**

#### Trigger: agent creation, after persona assignment

When a new agent is created and assigned a persona (via SessionCorrelator), the software scans `data/personas/{slug}/handoffs/` for existing handoff documents. This runs after persona assignment but before the agent greets.

#### Detection flow

1. Agent starts → `session-start` hook → SessionCorrelator assigns persona
2. Software lists `data/personas/{slug}/handoffs/*.md`
3. Sort by filename (timestamp prefix gives chronological order)
4. Take the **most recent 3**
5. Present as synthetic turns on the dashboard (see 0A.3)

#### Edge cases

| Scenario | Behaviour |
|---|---|
| No handoff directory | Skip — first-ever agent for this persona. No synthetic turns. |
| No `.md` files in directory | Skip — same as above. |
| Agent created by HandoffExecutor (`previous_agent_id` set) | **Still show the listing.** The operator may want to pick a different handoff than the automatic predecessor. The synthetic listing is informational — the operator decides. |
| Handoffs are weeks/months old | **No staleness threshold.** Show them. The operator can read the filenames (timestamps + summaries) and decide relevance. Filtering is a human judgement call. |

### 0A.3 Synthetic Injection Mechanism
- [x] **Decision: What is "synthetic injection" and how does it deliver software-generated turns into an agent's conversation?**

**Depends on:** 0A.2 (detection must run before injection)

**Context:** The operator explicitly requested a new delivery mechanism — not a hook response, not tmux send-keys. A software-generated "precursor turn" that appears in the agent's conversation before any human or agent interaction.

**Resolution:**

#### Dashboard-only display — the agent doesn't see it

Synthetic turns are **SSE events rendered on the dashboard** as conversation bubbles on the agent's card/panel. The agent never receives them. They're informational context for the operator.

This is option B from the seeded questions. No new tmux primitive, no hook changes, no Claude Code modifications. The dashboard already renders turn-by-turn content from SSE events — synthetic turns are a new SSE event type with a visual indicator marking them as system-generated.

#### Mechanism

1. Startup detection (0A.2) finds handoff files
2. `ChannelService` or a dedicated `HandoffDetectionService` emits an SSE event:
   - Event type: `synthetic_turn`
   - Data: `{agent_id, persona_slug, turns: [{type: "handoff_listing", filenames: [...], file_paths: [...]}]}`
3. Dashboard JS renders these as **visually distinct bubbles** (e.g., muted background, "SYSTEM" label, or a dashed border) before the agent's first real turn
4. Each listed handoff filename is a **copyable file path** — the operator clicks/taps to copy

#### What the operator sees

```
┌─────────────────────────────────────────────────────┐
│ ⚙ SYSTEM — 3 handoff documents available:           │
│                                                     │
│  1. 2026-03-02T23:39:00_epic9-workshop-restructure  │
│     _agent-id:1137.md                               │
│  2. 2026-03-02T19:28:59_epic9-section1-audit        │
│     _agent-id:1122.md                               │
│  3. 2026-03-02T04:58:10_org-workshop-section1       │
│     _agent-id:1117.md                               │
│                                                     │
│  Select a handoff to copy its path.                 │
└─────────────────────────────────────────────────────┘
```

#### General-purpose mechanism

Synthetic turns are reusable beyond handoff detection. Any software-generated context that the operator should see before the agent's first real interaction can use the same SSE event type. The `type` field in the turn data distinguishes use cases (`handoff_listing`, `context_briefing`, `system_notice`, etc.).

#### Ordering

Skill injection (tmux) → synthetic turns rendered on dashboard → agent greets. The synthetic turns don't block the agent's startup — they're dashboard-side only. The agent proceeds normally; the operator reads the listing and decides whether to intervene.

### 0A.4 Operator Rehydration Flow
- [x] **Decision: How does the operator tell an agent to rehydrate from a specific handoff?**

**Depends on:** 0A.3 (synthetic injection must present the list), 0A.2 (detection must find the handoffs)

**Context:** After the startup list is presented, the operator reviews the available handoffs and may choose to tell the agent to read one.

**Resolution:**

#### Manual: select, copy, paste

1. Agent starts → synthetic turn shows 3 most recent handoffs on dashboard
2. Operator reads the filenames (timestamps + summaries tell the story)
3. Operator clicks a handoff entry → file path copied to clipboard
4. Operator pastes the path into their message to the agent and adds context: *"Read the handoff at `{path}` and continue where they left off"*
5. Agent reads the file, absorbs the context, proceeds

This is exactly how the current handoff injection works today — the `compose_injection_prompt()` method produces a message like "Your predecessor wrote a handoff document... Read this file: `{path}`". The difference is the operator chooses which handoff (if any) rather than the software auto-injecting the most recent one.

#### Send paths

Any of the existing operator → agent paths work:
- **Dashboard respond box** — paste the path, type the instruction
- **Voice bridge** — *"Tell Robbo to read the handoff at [path]"*
- **Tmux bridge** — direct tmux send-keys (existing)

No standardised instruction format. The operator types naturally. Agents understand "read this file and continue" in plain English.

#### No "rehydrate" button in v1

A convenience button that auto-composes the injection prompt could be added later. For v1, copy-paste is sufficient — it's a startup-time operation, not a high-frequency action.

#### Multiple handoffs

The operator can tell the agent to read more than one. List shows 3; operator can paste multiple paths. No restriction.

#### No consumption tracking

Headspace does not track whether a handoff was "consumed" by a successor. The handoff file persists in the directory regardless. The `previous_agent_id` chain on Agent records provides the continuity trail; whether a specific handoff file was read is not tracked.

#### No card status change

The agent's dashboard card does not change after rehydration. The agent is just working — whether it read a handoff or started fresh is not a visible state distinction. The operator knows because they sent the instruction.

### 0A.5 On-Demand Handoff History
- [x] **Decision: How can the operator or agent query the handoff history for a persona?**

**Depends on:** 0A.1 (filename format determines what's in the listing)

**Context:** Beyond startup detection, the operator wants to be able to ask "what handoffs have happened for this persona?" at any time.

**Resolution:**

#### Three access paths, same data source

All paths read the filesystem: `data/personas/{slug}/handoffs/*.md`, sorted by filename (timestamp prefix).

| Path | Interface | Default |
|---|---|---|
| **CLI** | `flask org persona handoffs <slug>` | All handoffs, newest first |
| **API** | `GET /api/personas/<slug>/handoffs` | All handoffs, newest first, JSON |
| **Dashboard** | Persona panel/card → handoff history section | Most recent 5, expandable |

#### Per-entry information (parsed from filename)

| Field | Source |
|---|---|
| Timestamp | Filename prefix (ISO 8601) |
| Summary | Filename middle section (kebab-case slug) |
| Agent ID | Filename suffix (`agent-id:NNN`) |
| File path | Full absolute path (for copy-paste rehydration) |

No file content is read for the listing — the scannable filename (0A.1) provides enough context. The operator reads summaries from the directory listing and opens individual files only when needed.

#### Filesystem is the source of truth

The directory listing drives the display. The Handoff DB model carries `file_path` but is not the primary query source for this feature — the filesystem is authoritative. If a DB record references a file that's been moved/deleted, it won't appear in the listing. No reconciliation logic — the filesystem wins.

#### CLI options

```
flask org persona handoffs <slug>              # All handoffs, newest first
flask org persona handoffs <slug> --limit 5    # Most recent 5
```

Output: one line per handoff, showing timestamp, summary, agent ID, and path. Same scannable format as the directory listing.

### 0A.6 Handoff Instruction Changes
- [x] **Decision: What changes to the handoff instruction template enable the new filename format?**

**Depends on:** 0A.1 (filename format)

**Context:** The current `compose_handoff_instruction()` method tells the departing agent to write a 6-section document to a specific path. The filename is generated by the software with no summary component.

**Resolution:**

#### `generate_handoff_file_path()` changes

The method currently generates the full path: `{timestamp}-{agent_id_zero_padded}.md`

New behaviour: generate a **partial path with a placeholder** for the summary:

```python
def generate_handoff_file_path(self, agent: Agent) -> str:
    timestamp = datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%S")
    agent_tag = f"agent-id:{agent.id}"
    return str(
        project_root / "data" / "personas" / slug / "handoffs"
        / f"{timestamp}_<insert-summary>_{agent_tag}.md"
    )
```

The `<insert-summary>` placeholder is filled in by the departing agent.

#### `compose_handoff_instruction()` changes

Add explicit filename instructions to the existing template:

```
HANDOFF REQUESTED: Write a handoff document for your successor.

Write the document to: {file_path}

IMPORTANT — Replace `<insert-summary>` in the filename with a kebab-case
summary of your work (max 60 characters, no underscores). Example:
  2026-03-02T23:39:00_epic9-workshop-restructure_agent-id:1137.md

The document MUST include:
1. Current work: ...
(rest of existing 6-section template unchanged)
```

#### Polling thread fallback

The polling thread currently checks for the exact `file_path`. With the placeholder, the agent may produce a slightly different path. **Fallback:** if the exact path doesn't exist after the first poll, glob for `{timestamp}_*_{agent_tag}.md` in the handoff directory. The timestamp and agent ID portions are software-generated and deterministic — only the summary varies.

#### Document structure unchanged

The 6-section handoff document structure stays the same. Only the filename format changes.

### 0A.7 CLI & Organisation Workshop Integration
- [x] **Decision: What CLI commands support handoff continuity, and how do they integrate with the `flask org` CLI?**

**Depends on:** 0A.5 (handoff history query), Organisation Workshop Section 1.3 (CLI design)

**Context:** The Organisation Workshop (Section 1.3) defined `flask org` as the unified CLI with subgroups including `flask org persona`. Handoff history queries naturally live under the persona subgroup.

**Resolution:**

#### Namespace: `flask org persona handoffs`

Lives under `flask org persona` — handoffs are scoped to a persona, so they sit alongside other persona subcommands. No standalone `flask handoff` group.

#### Commands

```
flask org persona handoffs <slug>              # List all handoffs, newest first
flask org persona handoffs <slug> --limit N    # Most recent N
```

#### Output format

One line per handoff, plain text, designed for terminal scanning:

```
2026-03-02T23:39:00  epic9-workshop-restructure           agent-id:1137
2026-03-02T19:28:59  epic9-section1-audit                 agent-id:1122
2026-03-02T04:58:10  org-workshop-section1                agent-id:1117
```

Columns: timestamp, summary slug, agent ID. Full file path available with `--paths` flag (for copy-paste into rehydration instructions).

#### Data source

Filesystem only (`data/personas/{slug}/handoffs/*.md`). Same as 0A.5. No DB query.

#### No `--detail` flag in v1

Summarising handoff file content would require reading each file or an LLM call. The scannable filename makes this unnecessary for the common case — the operator reads the summary slug and opens the specific file if they want detail. A `--detail` flag that reads the first 5 lines of each file is a v2 convenience.
